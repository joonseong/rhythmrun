## 의사결정 로그

RunApp 고유의 리듬게임 컨셉(가속도계, BPM, 오디오 피드백)을 중심으로 기획하되, 러닝 트래커 요소(거리, 페이스, 소셜 공유)를 보조 기능으로 배치합니다. GPS 트래킹·코스 추천·소셜 피드 등 일반 러닝 앱 기능은 MVP OUT으로 분류하고, 리듬 러닝 경험 검증에 집중합니다.

## 2. MVP 성공 지표

> **v1.2 수정**: QA 지적 반영 — 재방문율 기준 통일(세션 완료 기준), 벤치마크 출처 명기, 보조 지표 2개 추가

### 주요 지표

| 지표 | 정의 | 목표 | 측정 기간 |
|------|------|------|----------|
| **재방문율** (Primary) | 첫 러닝 세션 완료 후 **7일 이내 2회 이상 세션을 완료**한 사용자 비율 | 40% 이상 | 출시 후 4주차 코호트 기준 |
| **리듬 피드백 활성화율** (Secondary) | 세션 중 오디오 피드백을 끄지 않고 세션을 완료한 비율 | 60% 이상 | 동일 코호트 기준 |
| **평균 세션 시간** (Secondary) | 완료된 세션의 평균 지속 시간 | 15분 이상 | 동일 코호트 기준 |

**[v1.2 변경 사유 — 재방문율 기준 통일]**  
기존 정의에서 기준점("첫 세션 완료")과 재방문 카운트 기준("세션을 시작")이 혼용되어 있었습니다. 세션을 시작만 하고 이탈한 경우를 재방문으로 집계하면 지표 품질이 저하되므로, 기준을 **세션 완료**로 통일합니다.
- 수정 전: "첫 러닝 세션 완료 후 7일 이내 **2회 이상 세션을 시작**한 사용자 비율"
- 수정 후: "첫 러닝 세션 완료 후 7일 이내 **2회 이상 세션을 완료**한 사용자 비율"

**[v1.2 변경 사유 — 보조 지표 추가]**  
재방문율 단일 지표만으로는 리듬 피드백이 실제로 사용되었는지 알 수 없습니다. 핵심 가치 가설("리듬 피드백이 재방문을 만든다")을 검증하기 위해 보조 지표를 추가합니다. 보조 지표 목표값은 초기 가정 기준이며, 출시 후 2주차 데이터를 기준으로 재조정합니다.

### 벤치마크 근거

| 항목 | 내용 |
|------|------|
| 비교 기준 | 피트니스 앱 평균 D7 재방문율 20-25% |
| 출처 | **[내부 가정 기준]** 공개된 피트니스 앱 리텐션 리포트(Adjust Mobile App Trends Report 2023, AppsFlyer State of App Marketing 2023)를 참고하였으나, 정확한 수치는 내부 가정으로 설정. 출시 전 리서치 단계에서 공식 수치로 업데이트 필요 |
| 목표 설정 근거 | 리듬게임 요소 추가로 평균 대비 2배 수준의 재방문율 달성 가설 |

### 판단 기준 (Decision Criteria)

| 결과 | 액션 |
|------|------|
| 재방문율 40% 미만 | 핵심 루프(리듬 피드백 → 재미 → 재방문) 재검토. 리듬 피드백 활성화율·평균 세션 시간 보조 지표와 교차 분석하여 문제 구간 특정 |
| 재방문율 40% 이상 | V2 기능 개발 진행. 보조 지표 기준으로 리듬 피드백 개선 우선순위 결정 |

## 3. MVP 범위 정의

### MVP IN (핵심 기능)

```
Priority 1 (Must Have)
├── 리듬 러닝 세션
│   ├── 가속도계 기반 걸음 감지
│   ├── 실시간 BPM 계산 및 표시
│   ├── BPM 연동 오디오 리듬 피드백
│   └── 기본 트래킹 (시간, 거리, 페이스)
│
├── 결과 화면
│   ├── 세션 요약 (리듬 점수, 거리, 칼로리)
│   ├── BPM 그래프 (시간대별)
│   └── 소셜 공유 (이미지 카드)
│
└── 사용자 계정
    ├── 회원가입 / 로그인
    └── 기본 프로필 (닉네임, 목표 페이스)
```

### MVP OUT (다음 버전)

```
V2 이후
├── 코스 추천 및 저장
├── 친구 팔로우 / 피드
├── 챌린지 시스템
├── 상세 칼로리 계산 (심박수 연동)
└── 음악 커스터마이징
```

> MVP는 **리듬 러닝 경험 자체**가 가치 있는지 검증하는 것이 목적

## 4. 구현 방식 명세

### ① 거리 측정 방식

> **v1.2 수정**: QA 지적 반영 — 보폭 계수 적용 근거 명기, 신장 미입력 케이스 처리 추가

- **MVP 채택 방식**: 가속도계 기반 보행 추정 (Step-based)
- **계산식**: `거리 = 감지된 걸음 수 × 사용자 보폭 추정값`

**보폭 추정 방식**

| 구분 | 계수 | 적용 BPM 구간 | 근거 |
|------|------|-------------|------|
| 걷기 | 신장(cm) × 0.413 | 140BPM 이하 | 일반적으로 인용되는 보행 보폭 추정 계수 범위(0.413~0.415) 하한 적용 |
| 러닝 | 신장(cm) × 0.46 | 141BPM 이상 | 러닝 보폭 추정 계수 범위(0.46~0.48) 하한 적용 |

**[v1.2 변경 사유 — 보폭 계수 분리]**  
기존 단일 계수(× 0.45)는 걷기·러닝 중간값으로 두 구간 모두에서 오차가 발생합니다. BPM 구간이 이미 운동 강도를 구분하고 있으므로, 동일 기준으로 보폭 계수를 분리 적용합니다. 적용 계수는 초기 가정 기준이며, **출시 전 별도 보폭 측정 테스트를 통해 검증 필요**합니다.

**신장 미입력 처리**

| 케이스 | 처리 방식 |
|------|------|
| 온보딩 중 신장 미입력 | 기본값 170cm 적용, 프로필에서 수정 가능 |
| 프로필 저장 중 신장 초기화 | 마지막 입력값 유지 |

### ② 칼로리 계산 방식

- **MVP 채택 방식**: MET(Metabolic Equivalent) 공식 기반 추정
- **계산식**: `칼로리(kcal) = MET × 체중(kg) × 운동 시간(시간)`

| 구간 | MET 값 |
|------|--------|
| 걷기 (140BPM 이하) | 3.5 |
| 러닝 (141~160BPM) | 7.0 |
| 고강도 (161BPM 이상) | 9.8 |

- 체중: 온보딩 시 입력. 미입력 시 기본값 65kg 적용 후 설정 유도
- 결과 화면 표기: "약 OOO kcal (추정값)" 형식으로 추정치임을 명시
- 심박수 연동 칼로리: V2에서 웨어러블 연동 시 정밀 계산으로 대체

**[v1.2 검토 항목 — BPM-MET 구간 정합성]**  
BPM 구간(140, 141, 161 기준)과 MET 적용 구간 정합성 재검토 필요. 현재 사용 계수는 일반 피트니스 표준이나, 러닝 시 가속도계 기반 BPM이 심박수 기반 운동 강도와 일치하는지 출시 전 검증 권고.

### ③ 소셜 공유 범위

- **MVP 채택 방식**: OS 기본 공유 시트(Share Sheet) 활용
  - iOS: UIActivityViewController
  - Android: Intent.ACTION_SEND
- **공유 콘텐츠**: 세션 결과 이미지 카드 (거리, 시간, 리듬 점수, 등급 포함)
- **특정 SNS 직접 연동**: MVP OUT (V2에서 Instagram Stories API 등 검토)
- **근거**: OS 기본 공유 시트를 활용하면 개발 범위를 최소화하면서 모든 SNS 플랫폼 지원 가능

> **v1.2 수정**: QA 지적 반영 — 온보딩 플로우 분기 추가, 목표 설정과 러닝 준비 역할 구분

```

    │
    ▼

    │
    ▼

    │
    ▼

    ▼                                              │
```

| 조건 | 분기 |
|------|------|

| 재방문 (토큰 만료) | 스플래시 → 로그인 |

**[v1.2 역할 구분 — 목표 설정 vs 러닝 준비]**
- **프로필 / 목표 설정**: 계정 단위 설정. 기본 목표 페이스(BPM), 체중, 보폭 등 지속 저장값 관리
- **러닝 준비 화면**: 세션 단위 설정. 오늘의 목표 거리, 이번 세션 BPM 조정, 리듬 모드 선택. 기본값은 프로필 설정값에서 불러오되 세션마다 변경 가능

```

[마이크만 거부]   → 제한 모드 안내 ("동작 감지 없이는 거리 측정 불가") + 재요청 옵션
```

**[v1.2 신규 추가 — 권한 재요청 정책]**

| OS | 정책 |
|---|---|
| iOS | 권한 거부 → 앱 내 재요청 1회 가능 → 이후 시스템 설정 유도 |
| Android | "다시 묻지 않음" 선택 전: 앱 내 재요청 1회 가능. 선택 후: 시스템 설정 유도 |
